# `development-plan.md` Review R1

## 1. 总体结论

这份开发计划的方向是对的，且相较原始架构稿已经明显收敛：

- 明确了 `raw` / `wiki` / `schema` / `state` 的分层
- 明确了 `skill` 负责编排、`tool` 负责确定性动作
- 引入了 `plan -> draft -> apply` 的显式链路，审计性比原方案更强
- 将 `index.md` 与 `page-index.json` 的职责分离，降低了“人类导航”和“机器检索”混用的复杂度

我的判断是：这份计划已经接近可实现，但还没有完全达到“可以直接开工”的程度。当前最大的缺口不是功能范围，而是几个关键语义还没有完全收口，尤其是 patch contract、失败恢复和幂等性。如果现在直接开始实现，后续大概率会在重试、回放、状态恢复这些工程细节上返工。

## 2. 我对这份方案的理解

这份方案本质上不是在做一个普通的问答机器人，而是在做一个“知识编译系统”：

1. `kb_source_add` 负责把原始资料纳入事实层。
2. `kb_plan_ingest` 负责根据 source 和当前 wiki 状态生成结构化变更计划。
3. `kb_draft_patch` 负责把计划渲染成可审计的知识变更草稿。
4. `kb_apply_patch` 负责以确定性方式将草稿落盘，并同步索引。
5. `kb_query` 则只消费已经沉淀到 wiki 的知识，不把临时推断冒充成知识库事实。

这个思路是成立的。它比“让 agent 直接编辑 Markdown 文件”强得多，因为它默认把变更流程拆成了可检查、可记录、可回滚的多个阶段。

## 3. 主要问题与建议

### 3.1 高优先级问题

#### 问题 1：`kb_draft_patch` 与 `kb_apply_patch` 的职责边界冲突

文档一方面说 `kb_draft_patch` 输出“完整 patch payload（含 markdown 正文）”，并且示例里已经把 `index.md` / `log.md` 的具体 append 内容放进 draft；另一方面又说 `kb_apply_patch` 中 `index.md` / `log.md` 的更新“在此步完成（确定性模板操作）”。

这两种说法不能同时成立，否则会出现以下问题：

- draft 到底是不是最终可执行产物，不清楚
- apply 到底是“纯执行器”还是“带有生成逻辑的执行器”，不清楚
- 审计链的最后一个“人工可 review 的稳定产物”不清楚

建议直接二选一，并写死：

- 推荐方案：`kb_draft_patch` 产出最终、完整、可回放的文件变更集合；`kb_apply_patch` 只做校验、写入、状态迁移和索引同步，不再生成任何正文或模板内容。

这个方案最符合当前文档强调的“LLM 产出物要持久化、可审计”的目标。

#### 问题 2：失败恢复语义没有闭合

当前设计要求：

- apply 前先检查 `git status --porcelain -- kb/`
- apply 时写入 `kb/state/runs/<run_id>_in_progress.json`
- apply 失败后保留 `in_progress` 标记
- 下次运行检测到残留 `in_progress` 时仅报 warning

这里存在明显冲突：

- 既然 `in_progress` 文件也在 `kb/` 下，那么失败后工作区通常已经不干净
- 下一次运行时，系统很可能在“看到残留 run marker 并尝试恢复”之前，就先被 dirty check 拒绝

也就是说，计划里虽然引入了事务标记，但没有真正定义“如何恢复未完成事务”。

建议补一节明确说明：

- dirty check 是否排除 `kb/state/runs/`
- 残留 `in_progress` 时是拒绝、提示 resume、还是允许 force-clear
- apply 部分成功时，如何判断哪些文件已经写入、哪些还没写入

如果不补这部分，MVP 很容易在第一次异常中断后进入“既不能继续、也不能安全重试”的状态。

#### 问题 3：`append` 型 patch 破坏幂等性

当前 draft 示例中，`index.md` 和 `log.md` 采用的是 `append`。

这会带来一个直接问题：如果 apply 过程中部分成功，而调用方重试 apply，那么这些内容很容易被重复写入，尤其是：

- `index.md` 出现重复链接
- `log.md` 出现重复日志项

但文档前面又把 tool 定义为“确定性、可测试、可幂等的动作”。这两者是冲突的。

建议将 `append` 改成幂等语义更强的操作，例如：

- `ensure_line`
- `upsert_entry`
- `replace_section`
- `insert_after_anchor_if_missing`

如果实现层仍坚持保留 `append`，那就必须明确要求 apply 端额外做重复检测和去重。

### 3.2 中优先级问题

#### 问题 4：patch 状态机有两个真相来源

当前状态同时由两部分表达：

- 文件所在目录：`plans/`、`drafts/`、`applied/`
- JSON 内字段：`status`

但文档没有定义哪个是权威。这样会产生几个后果：

- 文件被移动了但 `status` 没更新，怎么办
- `status=failed` 但文件还在 `drafts/`，系统如何区分“待应用草稿”和“失败草稿”
- 运维脚本应该信目录位置还是信 JSON 内容，不清楚

建议统一：

- 目录位置为唯一权威状态
- JSON 中的 `status` 只是冗余镜像，用于调试与展示

此外，`failed` 最好不要继续留在 `drafts/` 中。建议单独使用 `kb/state/failed/`，否则 drafted 与 failed 混在一起，会让后续工具逻辑变复杂。

#### 问题 5：schema grace period 缺少页面级迁移依据

当前文档提出：

- 仓库级 `schema/version.yaml`
- 不设页面级 `schema_version`
- 宽限期内旧页面 warning，过期后 error

但如果页面上没有任何迁移标记，lint 实际上很难稳健判断：

- 这是“符合旧规则、仍在宽限期内的页面”
- 还是“本来就不合法的坏页”

建议补一个轻量字段，不一定叫 `schema_version`，但至少需要有类似：

- `schema_last_migrated`
- `migrated_at`
- `page_format_version`

否则 migration grace period 在实现层会比较空泛。

#### 问题 6：`source_id` 前缀过短，缺少碰撞处理策略

当前 `source_id` 定义为 `src_sha256_<8位hash前缀>`。

对于 MVP 来说可以接受，但这份计划已经明确后续会面向更大规模知识库和团队协作。如果 `source_id` 被当作长期主键，那么 8 位前缀最好不要直接写死成最终规则。

建议至少写明其中一种策略：

- 默认 8 位，碰撞时自动扩展到 12 位或 16 位
- 或者从一开始固定为 12 位以上

这样能避免将来因为 ID 扩容而引入兼容问题。

## 4. 我认可的部分

以下设计我认为是正确且值得保留的：

### 4.1 `raw` 与 `wiki` 的边界清晰

`raw` 保留事实底座，`wiki` 承担知识沉淀，这个边界非常重要。否则系统很容易退化成“既存事实又存总结还夹杂临时推断”的混合仓。

### 4.2 `index.md` 与 `page-index.json` 分责合理

把“人类导航页”和“机器检索索引”拆开，是个正确决定。否则后面一旦检索需求增强，就会不断把机器字段塞回 Markdown 页面里，导致结构越来越怪。

### 4.3 MVP 收敛得比较克制

MVP 只做 source summary，不自动生成 concept / entity / analysis 页，这个收敛是对的。自动概念沉淀属于高价值但高风险能力，不适合在第一阶段引入。

### 4.4 把内容生成上升到 tool 层是正确修正

把 LLM 生成内容从 skill 层的 ad hoc 行为，提升成 `kb_draft_patch` 的显式持久化产物，是这份计划里最重要的修正之一。只要 contract 再收紧一些，这会成为整个系统审计链的核心。

## 5. 建议的最小修订项

如果你想尽快把这份计划推进到“可以开工”的状态，我建议至少补齐以下 5 点：

1. 明确 `kb_draft_patch` 与 `kb_apply_patch` 的职责边界，只保留一个地方负责生成最终文件内容。
2. 补写“失败恢复与重试语义”，尤其说明残留 `in_progress` 如何处理。
3. 将 `append` 改为幂等操作，或明确 apply 阶段必须去重。
4. 统一状态机的唯一真相来源，并把 `failed` 与 `drafted` 物理隔离。
5. 给 schema migration 增加页面级迁移标记，或删掉 grace period 机制，避免设计悬空。

## 6. 最终判断

这份开发计划已经具备比较强的架构方向感，主线没有问题，MVP 范围也控制得比较好。我不认为它需要推翻重写。

但按当前文本，我不会建议直接进入实现。原因不是“想得不够多”，而是有几处语义已经进入实现敏感区，再模糊下去，工程阶段就会自然分叉，最后不同模块会各自补自己的假设。

更准确地说，这份文档现在处于：

- 架构方向：成立
- MVP 范围：合理
- 工程约束：还差最后一轮收口

如果把前面列出的几个高优先级问题补清楚，这份计划就可以作为正式开发底稿使用。
