# OpenClaw KB 产品文档（用户/产品视角）

## 1. 产品目标

OpenClaw KB 的目标不是“临时检索文档”，而是把零散资料持续沉淀为可复用、可演进的知识资产：

- 让知识随时间“累积”，而不是每次提问都从原始材料重新拼接。
- 让用户专注在选材、提问和判断，知识整理与维护由 LLM 执行。
- 让知识库可追踪（有索引、有日志、有来源），便于长期维护和团队协作。

一句话定位：这是一个由 LLM 持续维护的个人/团队知识 wiki 系统。

## 2. 核心概念

### 2.1 三层结构

- `kb/raw`：原始资料层（只读事实来源，作为长期留存的“原件”）。
- `kb/wiki`：知识层（可编辑、可链接、可演进的 wiki 页面）。
- `kb/schema/`：内容规则层（约束 wiki 结构与 frontmatter 约定）。
- `skills/`：流程编排层（约束如何 ingest、query、lint，保证执行一致性）。

### 2.2 Wiki-first 工作方式

- 回答问题时优先使用 wiki 中已整理的知识，而不是直接回忆原文。
- 高价值问答可回写成 `analysis` 页面，形成“问答即资产”。
- 通过 `index.md`（导航）和 `log.md`（时间线）让知识库可浏览、可追溯。

### 2.3 持续维护而非一次性整理

- 每次新增资料会触发跨页面更新（sources/entities/concepts/index/log）。
- 定期 lint 检查结构和内容健康度，持续修复孤岛、缺链、矛盾和覆盖不足。

## 3. 主要功能

### 3.1 知识摄取（Ingest）

面向“新增资料并融入现有知识网络”的场景：

- 注册并读取新资料。
- 生成来源摘要页（`sources`）。
- 创建或增量更新实体页（`entities`）与概念页（`concepts`）。
- 更新索引与变更日志，确保可导航、可审计。
- 发现冲突时显式标注，避免隐式覆盖旧结论。

### 3.2 知识问答（Query）

面向“基于已有知识做判断与决策”的场景：

- 在 wiki 内检索并精读相关页面后回答。
- 回答强调溯源（引用相关 wiki 页面/来源）。
- 信息不足时明确指出缺口，不凭空补全。
- 深度问答可沉淀为分析页（`analyses`），并记入日志。

### 3.3 知识体检（Lint）

面向“知识库质量治理”的场景：

- 检查孤岛页面、幽灵链接、缺失交叉引用、stub 堆积。
- 发现矛盾、过时内容、覆盖缺口与可外部检索的数据缺口。
- 输出按严重性分级的报告，支持后续有选择地修复。
- 每次完整 lint 结果（包括无问题）都会记录到日志，便于长期跟踪。

### 3.4 变更可追踪

- `index.md` 提供结构化导航。
- `log.md` 提供时间序列变更记录（ingest/query/lint）。
- 便于回看“知识是如何形成的”，而不仅是“当前答案是什么”。

## 4. 典型使用流程

### 4.1 单篇资料接入流程（最常见）

1. 用户提供一篇新资料并发起 ingest。
2. 系统生成来源摘要，并识别需创建/更新的实体与概念页面。
3. 用户确认更新计划后执行写入。
4. 系统更新索引与日志，形成完整可追溯闭环。

典型产出：
- 1 个 `source` 页面
- 若干 `entity/concept` 页面更新
- `index.md` 新增入口
- `log.md` 新增本次 ingest 记录

### 4.2 问题分析沉淀流程

1. 用户提出问题（如对比、取舍、边界判断）。
2. 系统先检索并阅读 wiki 相关页面，再给出结构化结论。
3. 若结果有长期价值，用户可确认保存为 `analysis` 页面。
4. 系统同步更新索引与日志。

### 4.3 周期性健康治理流程

1. 用户发起 lint。
2. 系统给出错误/警告/建议分级发现。
3. 用户选择需要修复的项。
4. 系统执行修复并记录本次 lint 结果。

## 5. 当前支持范围与限制（基于当前仓库状态）

截至仓库当前状态（2026-04-19 刷新快照）：

### 5.1 当前支持

- 以 V2 架构为主：围绕 ingest/query/lint 三大流程稳定运行。
- 已有完整 wiki 结构：`sources`、`entities`、`concepts`、`analyses`、`reports`。
- 支持通过日志追踪“新增资料、问答沉淀、健康检查”的演进过程。

### 5.2 当前限制

- 新资料注册原生支持 Markdown / plaintext，并可在安装 Python MarkItDown 后转换 HTML、CSV、JSON、XML、PDF、DOCX、PPTX、XLSX/XLS、EPUB；ZIP、OCR/图片、音频转录、Outlook/email、YouTube、SVG 与 plugins 暂不支持。
- 资料外部事实不会在 lint 中自动补齐；仅标注“可通过外部检索补充”的缺口。
- 对 `kb/wiki` 的多文件修改需遵循既定治理流程（plan → draft → apply）。
- Obsidian GUI 侧的可视化兼容性 spot-check 仍有待完成（文档中已标注 pending）。
- 历史样例尚未全部回填 raw source 与 manifest；溯源完整性以具体 source 是否已有 manifest 为准。

### 5.3 使用注意

- `kb/raw` 是原始事实层，不应被改写。
- 回答应以 wiki 为第一信息源；缺信息时应明确“不足”而非猜测。
- 每次关键操作应保持索引与日志同步更新，避免知识“写入但不可发现”。
- 如涉及提交操作，执行前应确认暂存区范围，避免把无关改动一并提交。

## 6. 适合与不适合的场景

### 6.1 适合

- 需要长期累积知识的个人学习与研究（持续读材料、持续沉淀）。
- 团队知识管理（会议纪要、调研结论、项目知识的结构化沉淀）。
- 需要“可追溯结论”的分析工作（能回到具体页面与来源）。
- 同一主题会被反复提问、反复演化的场景。

### 6.2 不适合

- 只做一次性问答、无长期沉淀需求的轻量场景。
- 需要即时覆盖大量非文本格式来源且不允许前处理的场景。
- 无法接受“信息不足时明确留白”的场景（本产品强调证据边界）。
- 希望完全自动化替代人工判断冲突结论的场景（冲突裁定仍需人参与）。

## 7. 面向使用者的价值总结

OpenClaw KB 提供的核心价值是：把“聊天式答案”升级为“可持续维护的知识资产”。  
用户得到的不仅是一次回答，而是一个随着资料和问题不断变强的知识系统。
